# 26 — Sprint Process

## Purpose

The sprint process turns strategy into controlled execution.

## Sprint Inputs

A sprint may include product specifications, engineering tasks, design work, bugs, technical debt, research, documentation, and release work.

## Planning Requirements

Every sprint item must have:

- Owner
- Priority
- Scope
- Acceptance criteria
- Design or specification link when applicable
- Dependencies
- Risk notes
- Definition of done

## Execution Rhythm

Daily execution should focus on blockers, risk, scope control, and customer impact. Work should be visible, documented, and updated as facts change.

## Scope Control

If scope changes materially, update the specification or create a follow-up item. Hidden scope expansion creates unreliable delivery.

## Review

Sprint review should inspect shipped work, missed work, quality, incidents, customer impact, and follow-up actions.

## Retrospective

Every sprint should improve the operating system. Process lessons must become documentation, checklist changes, or architecture improvements.

## Success Criteria

A successful sprint ships meaningful progress without sacrificing quality, security, documentation, or trust.
